### Model Descriptions

This document defines the available models provided by the `openai` provider: **GPT-4.1 nano (small)**, **GPT-4.1 mini (medium)**, and **GPT-4.1 (large)**. It outlines their capabilities, limitations, and recommended use cases to guide an automated model selection process based on task complexity, performance needs, and inferred computational cost. The primary directive is efficiency: **always default to the Nano (small) model unless task characteristics explicitly require escalation.** Escalation should be triggered by specific indicators of complexity exceeding the capabilities of the more economical option.

All models feature a refreshed knowledge cutoff of June 2024 and support context windows up to 1 million tokens. Computational cost increases significantly with model size and context length.

**Core Selection Principle:** The selection logic should prioritize Nano > Mini > Large. Escalate only when the query's characteristics demonstrably match the capabilities unique to the next tier up.

---

#### OpenAI GPT-4.1 nano (Small Model) - *Default Choice*

```json
{
  "provider": "openai",
  "model": "small"
}
```

**Provider**: openai
**Model Identifier**: `gpt-4.1-nano` (Example API identifier; use actual deployment name)

**Description**:
The smallest, fastest, and most computationally efficient model. Optimized for speed and low resource usage, it handles many common tasks effectively, including code exploration and structural analysis. Features a 1M token context window. **This is the primary default model.**

**Strengths**:
- **Lowest Cost & Highest Speed**: Most efficient in terms of resource usage and latency.
- **Code Exploration**: Effective for searching codebases, locating implementations, and describing code structure.
- **Efficient Large Context**: Suitable for searching or retrieving specific information from large inputs efficiently.
- **Strong Baseline**: Capable performance on straightforward requests, classifications, and formatting.

**Use Cases**:
- Searching codebases ("Where is X implemented?", "Show me the definition of Y").
- Describing code structure or function implementation.
- Answering basic questions about component interactions ("How does A call B?").
- Simple code formatting or basic refactoring suggestions.
- Classification or categorization.
- Autocompletion.
- Simple data extraction.
- Request routing based on keywords or intent.

**Limitations**:
- Not suited for tasks requiring deep diagnostic reasoning or understanding complex causality.
- Struggles with multi-step planning or complex procedural generation.
- Less capable for complex code generation or debugging subtle issues.

**Selection Guidance**:
**Select by default for nearly all incoming queries.** Utilize Nano for exploratory tasks, code searching/description, structural questions, classification, autocompletion, simple generation, and formatting. Its efficiency makes it the preferred choice. **Escalate to Mini *only if* the query's characteristics clearly signal a need for diagnostic reasoning (e.g., asking "why" about complex failures), multi-step planning, or analysis of complex interactions beyond simple structural understanding.**

#### OpenAI GPT-4.1 mini (Medium Model) - *For Complex Tasks*

```json
{
  "provider": "openai",
  "model": "medium"
}
```

**Provider**: openai
**Model Identifier**: `gpt-4.1-mini` (Example API identifier; use actual deployment name)

**Description**:
A highly capable model offering a significant increase in reasoning and coding abilities over Nano. It excels at tasks requiring deeper analysis, debugging, and understanding complex interactions, representing a substantial step up in computational cost and capability.

**Strengths**:
- **Advanced Reasoning**: Handles diagnostic, planning, and complex reasoning tasks more effectively than Nano.
- **Stronger Coding/Debugging**: More proficient at identifying bugs, understanding interaction effects, and generating/refactoring complex code.
- **Complex Instruction Following**: More reliable for intricate, multi-part instructions.
- **Deeper Long Context Analysis**: Can reason more effectively across large contexts to synthesize complex insights.

**Use Cases**: **(Select only when Nano's limitations are apparent for the query)**
- Diagnosing complex bugs involving interactions ("Why is function X failing when called by service Y under Z condition?").
- Planning multi-step implementation approaches for moderately complex features.
- Analyzing system behavior to identify subtle issues or non-obvious interaction effects.
- Refactoring code with complex dependencies or logic.
- Reasoning across multiple documents/code sections to synthesize complex insights (beyond simple retrieval).
- Assisting with debugging traces or explaining intricate algorithms.

**Limitations**:
- Significantly higher computational cost and latency compared to Nano.
- Inefficient for tasks Nano can handle adequately.

**Selection Guidance**:
**Select Mini *only when* the query demonstrably requires capabilities exceeding Nano.** Trigger selection for Mini if the task involves deep diagnostic analysis (especially "why" questions about complex failures), requires planning multiple implementation steps, needs analysis of subtle interaction effects, or involves complex code generation/debugging clearly beyond Nano's scope. **Avoid selecting Mini for tasks primarily focused on search, structural description, or simple generation/formatting.** Escalation from Mini to Large signals detection of exceptionally high complexity.

#### OpenAI GPT-4.1 (Large Model) - *Exceptional Use Only*

```json
{
  "provider": "openai",
  "model": "large"
}
```

**Provider**: openai
**Model Identifier**: `gpt-4.1` (Example API identifier; use actual deployment name)

**Description**:
The apex model with state-of-the-art performance across all capabilities. It represents the **highest tier of computational cost and capability**. Its selection should be triggered **extremely rarely**, only by tasks demanding the most profound level of reasoning and strategic planning.

**Strengths**:
- **Peak Performance**: Highest scores across all benchmarks.
- **Deepest Strategic Reasoning**: Unmatched for high-level architectural design, complex system analysis, and strategic planning.

**Use Cases**: **(Select ONLY in extremely rare cases of highest complexity)**
- Generating foundational system architecture designs for novel, large-scale systems.
- Devising strategic plans for major, cross-cutting refactoring efforts across vast codebases.
- Performing deep analysis requiring synthesis of information across extremely large, complex datasets where Mini is predicted to be insufficient.
- Addressing tasks equivalent in complexity to principal-level architectural or research problems involving high ambiguity and abstract reasoning.

**Limitations**:
- **Highest Computational Cost & Latency**: Resource-intensive and slow relative to Nano/Mini.
- **Massive Overkill**: Grossly inefficient for anything other than its narrow band of intended high-level strategic tasks.

**Selection Guidance**:
**Select Large *only as a final resort* when task analysis indicates complexity corresponding to foundational architectural design, highly complex multi-system strategic planning, or synthesis requirements demonstrably beyond Mini's capabilities.** This selection signifies the detection of the absolute highest level of task complexity, ambiguity, and reasoning depth, where lower-tier models are predicted to fail comprehensively. Its selection should be an indicator of an exceptional query requiring maximum computational effort.
